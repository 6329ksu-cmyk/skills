# User Cycle Preferences

This skill was created from a conversation where the user explicitly preferred long autonomous cycles over frequent interviews.

Preserve these behaviors:

- Do not keep asking clarifying questions.
- Infer the goal, make reasonable assumptions, and spend a substantial pass improving the artifact.
- The user wants to react after a full cycle, not manage each step.
- The user may give coarse feedback such as "wrong", "too weak", "make it stricter", or "continue"; treat that as the next cycle's steering input.
- The cycle should include gap-finding and gap-filling, not only review.
- Briefings must be short enough to read quickly and focused on judgment: what changed, why, tradeoffs, validation, remaining risk, and next recommendation.
- Avoid dumping unrelated tech stack details, file lists, or long context that does not help the user decide.
- Use human judgment checkpoints mainly at the end of a cycle. Mid-cycle questions are for blockers only.

The intended feeling is: "Codex poured tokens into one long pass, came back with a clear result, and I can reject or steer it in one sentence."
